# IoLIFE API Specification (Current `main.py` Implementation)

This document describes the API behavior implemented in `main.py`.
It is a practical spec focused on reproducible requests.

## 1. Scope

- Target host: `app.iolife.toshiba-lifestyle.com`
- API version path: `/v1`
- Client type: Toshiba/Midea IoLIFE-compatible flow
- Implemented commands:
  - `login`
  - `devices`
  - `call`
  - `transparent`
  - `ac-query`
  - `ac-power`

## 3. Common Request Format

All signed POST calls use `application/x-www-form-urlencoded`.

Common parameters:

- `format=2`
- `stamp=yyyyMMddHHmmss` (local time)
- `language` (default: `ja_JP`)
- `appId=1203`
- `src=203`
- `sessionId` (required after login for session endpoints)

## 4. Signature Algorithm

### 4.1 Path

- Request path format: `/{SERVER_VERSION}/{endpoint}`
- Example: `/v1/user/login`

### 4.2 Canonicalization

1. Sort all request params by key.
2. Build query-like string: `k1=v1&k2=v2&...`.
3. Concatenate: `path + payload + APP_KEY`.
4. SHA256 hex lower-case.
5. Send as `sign`.

Pseudo:

```text
sign = sha256_hex(path + "&".join(sorted(k=v)) + APP_KEY)
```

## 5. Authentication Flow

### 5.1 `user/login/id/get`

- Method: `POST`
- Endpoint: `/v1/user/login/id/get`
- Signed: yes
- Required params:
  - common params
  - `clientType`
  - `loginAccount`

Expected response:

- `errorCode == "0"`
- `result.loginId` present

### 5.2 `user/login`

- Method: `POST`
- Endpoint: `/v1/user/login`
- Signed: yes
- Required params:
  - common params
  - `clientType`
  - `loginAccount`
  - `password`
  - `iampwd`
  - `pushType`
  - `pushToken`

Field derivation:

- `password_sha = SHA256(rawPassword)`
- `password = SHA256(loginId + password_sha + APP_KEY)`
- `iampwd = SHA256(loginId + MD5(MD5(rawPassword)) + APP_KEY)`

Response fields used:

- `result.sessionId`
- `result.userId`
- `result.accessToken`
- `result.randomData`

Local decode behavior:

- `dataKey = AES_decrypt(accessToken, md5(APP_KEY)[:16], ECB)`
- `dataIV = AES_decrypt(randomData, md5(APP_KEY)[:16], ECB)` if available

## 7. Endpoint Specifications

### 7.1 Devices

- Method: `POST`
- Endpoint: `/v1/appliance/user/home/page/list/info`
- Signed: yes
- Required params:
  - common params + `sessionId`
  - `appVersion`
  - `clientType`
  - `appId`

### 7.2 Transparent Send

- Method: `POST`
- Endpoint: `/v1/appliance/transparent/send`
- Signed: yes
- Required params:
  - common params + `sessionId`
  - `applianceId`
  - `funId=0008`
  - `order` (encrypted string)

### 7.3 Generic Call

- Method: `POST`
- Endpoint: `/v1/{endpoint}`
- Signed: yes
- Required params:
  - common params
  - optional `sessionId`
  - user-provided key-values

## 8. `order` Generation for Transparent API

`order` generation path (`payload-hex` mode):

1. Build WiFi datagram from payload (`build_wifidatagram`).
2. Convert bytes to signed decimal CSV (`bytes_to_dec_string`).
3. AES encrypt CSV with `dataKey` and optional `dataIV`.
4. Hex output becomes `order`.

### 8.1 WiFi Datagram Layout

Header + body format used by `build_wifidatagram`:

- `0-1`: `0x5A 0x5A`
- `2`: version (`0x01`)
- `3`: flags (`0x00`)
- `4-5`: total length (little-endian)
- `6-7`: `msgType` (default `32`)
- `8-11`: `msgId` (incremental)
- `12-19`: timestamp bytes
- `20-25`: appliance ID low 6 bytes (little-endian)
- `26-27`: response timeout
- `28-33`: channel id (zeros)
- `34-39`: reserve (zeros)
- `40..`: payload bytes
- last 16 bytes: signature area (zeros)

## 9. Reply Decode

If `--decode-reply` is set and `errorCode == "0"`:

1. decrypt `result.reply` with session `dataKey`/`dataIV`
2. decimal CSV -> bytes
3. parse with `parse_wifidatagram`
4. print:
  - `replyDec`
  - `replyParsed` (`length`, `msgType`, `msgId`, `devIdHex`, `bodyHex`, `rawHex`)

## 10. AC (`0xAC`) Helper Protocol

### 10.1 UART Frame (Inner Payload)

Generated by `build_ac_uart_payload`:

- Header: `55 AA CC 33`
- Length: little-endian
- Version/device: `01 AC`
- Type:
  - query: `0x0003`
  - control: `0x0002`
- Tail: CRC16-CCITT

### 10.2 CRC

- `crc8_854`: table-based checksum
- `crc16_ccitt`: polynomial `0x1021`

### 10.3 `ac-query`

- `query_type=all`:
  - body length 22
  - starts with `41 81 00 FF ...`
  - random byte + CRC8
- `query_type=<property>`:
  - starts with `B1 01 <property_code> 00`
  - random byte + CRC8

Property codes are implemented in `AC_QUERY_PROPERTIES`:

### 10.4 `ac-power`

`state=on|off` is encoded as:

- body prefix: `B0 01`
- TLV: `01 00 01 <state>`
- random byte + CRC8
- wrapped with request type `0x0002`

## 11. Error Behavior

### HTTP errors

- `HTTP 504` is wrapped with hint:
  - cloud reachable, device response timeout
  - likely offline/wrong appliance-id/invalid payload

### API error response

- `_require_success` treats non-zero `errorCode` as failure.
- Some endpoints may return alternative shape:
  - example: `{"code":"","message":"invalidSession"}`
  - this is not normalized by `_require_success`; caller should handle it.

## 12. CLI to API Mapping

- `login` -> `user/login/id/get`, `user/login`
- `devices` -> `appliance/user/home/page/list/info`
- `call` -> custom endpoint
- `transparent` -> `appliance/transparent/send`
- `ac-query` -> local AC payload builder + `transparent`
- `ac-power` -> local AC control payload builder + `transparent`

## 13. Security Notes

- Credentials and session tokens are sensitive.
- `session.json` includes session/auth data in plaintext.
- Use only for legal/authorized research environments.
